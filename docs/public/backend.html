Chapter 00 · The brief

Four apps, one core.

A backend for a Malaysian e-hailing system serving four audiences — rider, driver, driver-web (onboarding portal), and admin — built as a modular monolith on Postgres and Redis, with Auth0 in front. One backend, one database, four frontends. Designed to ship a thin v0.1 today and grow into v1.0 without restructuring.

Chapter 01 · Topology

The shape of the system, at a glance.

Four frontends speak to one backend over versioned, audience-scoped API prefixes. Auth0 sits in front for phone OTP and admin SSO; the backend verifies tokens against JWKS. Postgres holds the schema; Redis carries hot real-time state and the BullMQ queue.

src/app.ts ts 
app.register(auth0Verify);                                                 // JWKS-cached, role from claim

app.register(riderRoutes,     { prefix: '/api/v1/rider',      preHandler: requireRole('rider') });
app.register(driverRoutes,    { prefix: '/api/v1/driver',     preHandler: requireRole('driver') });
app.register(driverWebRoutes, { prefix: '/api/v1/driver-web', preHandler: requireRole('driver') });
app.register(adminRoutes,     { prefix: '/api/v1/admin',      preHandler: requireAdminRole() });

app.register(webhookRoutes,   { prefix: '/api/webhooks' });   // signature-verified, no auth
app.register(publicRoutes,    { prefix: '/api/public' });     // trip-share, content, cancel reasons

io.of('/ws/rider').use(socketAuth('rider'));
io.of('/ws/driver').use(socketAuth('driver'));
io.of('/ws/public/:share_token').use(shareTokenAuth());                  // no app required

Chapter 05 · House rules

Eleven rules to keep the monolith honest.

A modular monolith degrades into a regular monolith the day someone reaches across a boundary "just this once". These are the fences.

01

One service per module is the source of truth.

Routes, workers, and websockets all delegate to the module's service. Repos stay private to their module — no cross-module SQL.

02

Routes by audience, domain by context.

api/{rider,driver,driver-web,admin}/ hold the per-audience surface; modules/ hold the domain. Routes are thin and call into services. DTOs live with routes; domain types live with modules.

03

State changes go through the state machine.

No raw UPDATE trips SET status, no raw UPDATE driver_applications SET state. Every transition writes to trip_events or application_steps. audit_log and trip_events are append-only — Postgres triggers reject UPDATE/DELETE, and a nightly worker links rows by prev_hash.

04

Dispatch and tracking have no HTTP routes.

Dispatch is queue-driven (BullMQ); tracking is WebSocket-driven (Redis GEO). Different scaling profiles, different transports.

05

Workers run in a separate process.

The API and the worker share code, not a runtime. Scale where it's needed without touching request latency.

06

Money in integer cents, geometry in 4326, time in Asia/Kuala_Lumpur.

amount_cents int · GEOGRAPHY(POINT, 4326) · ISO Mon-Sun weekly bins computed in KL local. Floats compound rounding errors; floating timezones break finance.

07

One Postgres database, three roles.

app_owner for migrations. app_user at runtime. app_readonly for BI. The backend is the trust boundary.

08

Design for extraction, not separation.

Modules look like services from outside (one entrypoint, owned data) so any one — likely tracking first — can be lifted into its own process when it earns it.

09

Auth0 owns identity. The backend owns roles and audit.

No auth_otp_codes, no refresh_tokens tables. Frontends carry Auth0 access tokens; the backend verifies via JWKS (cached, pre-warmed) and reads roles from a custom claim. Local users are linked by external_identities (provider, provider_sub) — one local user can carry multiple Auth0 subs.

10

Compliance is a first-class module.

PDPA consent, audit trail, SAR / erasure, EVP per-vehicle (APAD/LPKP), PSV per-driver, and per-trip PA insurance certificates are modeled now — even where the v0.1 workflow is manual. Don't scaffold them last.

11

Stubs match real shape, or they don't ship.

The v0.1 stub keeps Postgres, Redis, Auth0, Socket.io, and Maps real; payments, dispatch, OCR, surge, notifications, insurance, voice, SOS are stubbed. A stub that lies — returning status='stubbed' synchronously, skipping trip_offers events, hiding the surge badge — is a bug. Frontends should not be able to tell the difference.
